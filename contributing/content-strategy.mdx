# Content Strategy

A5SYNC relies heavily on clear, scannable, and actionable documentation. When documenting features across our various portals (Storefront, Reseller Portal, etc.), it's vital to follow a consistent pacing strategy.

## The "One Read" Philosophy
Every page in the A5SYNC docs should represent **one specific concept or user goal**. A reader should be able to open a page, understand the immediate context, achieve the goal, and leave.

**A good read should contain:**
1. **A clear title & description:** Set expectations immediately in the YAML frontmatter.
2. **Context (1-2 sentences):** Explain *why* the user needs this feature or *what* it does.
3. **Prerequisites (if any):** Does the user need admin permissions? State it early.
4. **Actionable Steps:** Ordered lists outlining exactly what to click.
5. **Expected Outcome:** A brief description or screenshot of what success looks like.

## Page Length & Sections

### Conceptual Breaking Points (When to Split)
When documenting a massive domain-level feature (like **Orders**, **Quotes**, or **Product Catalogs**), do not cram everything into one page. 

**How to determine when to break:**
1. **The "CRUD" Test:** If a feature involves Creating, Reading, Updating, and Deleting complex entities, those usually deserve separate pages or highly distinct tabs.
2. **The "Persona" Test:** Does configuring the feature require an Admin, but using it only requires a standard User? Split the "Configuration" into a separate page from the "Usage".

**Example Breakdown for `Orders`:**
* **Bad:** A single `orders.mdx` that explains what an order is, how to configure fulfillment stages, how to issue refunds, and how to query the orders API.
* **Good:** Create a folder `user-guides/storefront/orders/` and break it conceptually:
  - `introduction.mdx`: What is the Order module? High-level overview.
  - `fulfillment-stages.mdx`: Configuration and definitions of the states.
  - `returns-and-refunds.mdx`: Purely the workflow for handling refunds.

### Grouping in `docs.json`
When you break a conceptual topic down like the `Orders` example above, encapsulate it cleanly in `docs.json`:
```json
{
  "group": "Order Management",
  "pages": [
    "user-guides/storefront/orders/introduction",
    "user-guides/storefront/orders/fulfillment-stages",
    "user-guides/storefront/orders/returns-and-refunds"
  ]
}
```

## Tone and Formatting Guidelines
- **Use Active Voice:** Start action steps with a verb. ("Click the Save button" > "The Save button should be clicked").
- **UI Elements:** Always **bold** UI elements. (e.g., Go to **Settings** > **Billing**).
